## Overall Impression

The component architecture is the strongest thing here — ViewParameters + ViewEditPage + useAdminEdit compose cleanly. Problems are UX-layer: 10% width cards are layout-broken, delete bypasses the design system with native confirm(), mixed German/English is jarring, and there's a silent data-loss bug in Work Item Type where Sort Order is never saved.

## What's Working

1. Exemplary component reuse — ViewParameters + ViewEditPage + useAdminEdit means adding a new entity type is <30 lines of code.
2. Config page UX is noticeably better — SettingsSection + SettingsRow with hints, save loading state, toast feedback, try/catch.
3. Color token discipline — no rogue hex values, no box-shadow, no gradient text. Tonal layering intact.

## Priority Issues

**[P1] Native confirm() for deletion breaks the design system** — window.confirm() produces OS-native browser dialog with no styling, contradicts "Predictable, not surprising" principle. Also: delete call is not awaited before refresh() in brand/category/device/work-item-type (race condition). Fix: replace with inline confirmation or popup component; await the $fetch delete call.

**[P1] width: 10% entry cards are layout-broken** — At 1440px viewport, cards are ~127px wide. Work Item Type renders 8 fields at that width. Fix: clamp(200px, 20%, 340px) or grid with minmax(240px, 1fr).

**[P1] Create button sits after the list** — Admin convention is top-right adjacent to page header. Fix: add a named #actions slot to CommonPage, move create button there.

**[P1] Silent data loss: Work Item Type Sort Order never saved** — fields[5] (Sort Order) skipped in save(). Every Sort Order edit silently discarded. Fix: add sortOrder: Number(page.value.fields[5]?.value) to the saveToApi call.

**[P2] Mixed German/English fractured language register** — Titles English, confirmations German, buttons English. Fix: standardize to German for all user-facing strings.
